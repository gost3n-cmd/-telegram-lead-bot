# Архитектурные решения

## Решения, принятые при реализации

### D001: Модель Lead — pydantic BaseModel
- **Дата:** 2026-06-22
- **Контекст:** Архитектура (раздел 10) указывает поля Lead, но не указывает конкретную библиотеку для модели. Раздел 11 упоминает `pydantic` 2.x.
- **Решение:** Lead реализован как `pydantic.BaseModel` с полной валидацией типов.
- **Обоснование:** Архитектура явно указывает pydantic в разделе 11. BaseModel обеспечивает валидацию, сериализацию и читаемое представление.

### D002: SaveResult — простой класс с __slots__
- **Дата:** 2026-06-22
- **Контекст:** Раздел 10 указывает `SaveResult(success: bool, error: str | None)`. Не указывает форму (dataclass / pydantic / NamedTuple).
- **Решение:** SaveResult реализован как минималистичный класс с `__slots__` и `__bool__`. Не наследует BaseModel — у него нет валидируемых полей со сложными типами.

### D003: StorageInterface — ABC (Abstract Base Class)
- **Дата:** 2026-06-22
- **Контекст:** Раздел 10 описывает контракт хранилища как абстрактный интерфейс. Не указывает механизм (ABC vs Protocol).
- **Решение:** Использован `abc.ABC` + `abc.abstractmethod` — классический подход, совместимый с любой версией Python.
- **Обоснование:** Проще для статической проверки импортов, понятен при чтении кода.

### D004: email_exists() внутри критической секции лока
- **Дата:** 2026-06-22
- **Контекст:** Раздел 14.3 упоминает два варианта: отдельный load_workbook или вход в ту же критическую секцию.
- **Решение:** `email_exists()` также входит в критическую секцию `asyncio.Lock` (тот же лок, что и `save_lead`).
- **Обоснование:** Это устраняет TOCTOU-гонку между проверкой и сохранением при конкурентных запросах одного email. Подтверждено тестом `test_concurrent_saves_no_lost_update` (10 конкурентных сохранений → ровно 10 записей).

### D005: Путь к xlsx — относительно рабочего каталога процесса
- **Дата:** 2026-06-22
- **Контекст:** Раздел 16 указывает `EXCEL_FILE_PATH` по умолчанию `data/ai_content_os.xlsx`.
- **Решение:** Конфигурация принимает путь как есть. ExcelStorage создаёт родительский каталог при необходимости через `os.makedirs`.

### D006: LEADS_SHEET_NAME — конфигурируемый параметр
- **Дата:** 2026-06-22
- **Контекст:** Раздел 16 делает `LEADS_SHEET_NAME` конфигурируемым через `.env`.
- **Решение:** Реализовано через pydantic-settings как часть `BotConfig`.

### D007: build_system.py — не изменён в рамках данной итерации
- **Дата:** 2026-06-22
- **Контекст:** Раздел 4 описывает Sheet Ownership для генератора, но раздел 4.7 указывает, что конкретная реализация — задача Этапа 2.
- **Решение:** Файл скопирован в `generator/build_system.py` без изменений. Sheet Ownership будет реализован в отдельной итерации.
- **Обоснование:** Архитектура явно указывает, что конкретная реализация шага 4.3 — задача Этапа 2 (см. раздел 12).

### D008: ExcelStorage создаёт файл при отсутствии
- **Дата:** 2026-06-22
- **Контекст:** Раздел 4.5 архитектуры: "Файла нет на диске → поведение как раньше". Раздел 7: ExcelStorage "находит/создаёт лист Leads".
- **Решение:** `_load_workbook_sync` проверяет существование файла; если файл отсутствует — создаёт новый `Workbook()` и родительский каталог.
- **Обоснование:** Соответствует разделу 4.5 — бот создаёт лист Leads при первом сохранённом лиде. Подтверждено тестом `test_save_lead_creates_file_when_missing`.

### D009: DI через closure (фабрику роутера)
- **Дата:** 2026-06-22
- **Контекст:** Раздел 5/9 архитектуры: Application зависит от интерфейса (Domain), а не от конкретной реализации (Infrastructure). Конкретная реализация подставляется один раз — в точке композиции (`main.py`).
- **Решение:** `create_email_router(lead_service)` — фабрика, принимающая `LeadService` через closure. `main.py` передаёт готовый `LeadService` в роутер при сборке.
- **Обоснование:** Гарантирует, что handler не знает о конкретной реализации хранилища. Вариант через `dp["lead_service"]` (kwargs resolver) также допустим, но closure делает зависимость явной и проверяемой типизацией.

### D010: openpyxl I/O выполняется в ThreadPoolExecutor
- **Дата:** 2026-06-22
- **Контекст:** Раздел 14.4: `load_workbook` и `wb.save` — синхронный disk I/O, занимают event loop.
- **Решение:** Все синхронные openpyxl-операции оборачиваются через `loop.run_in_executor(self._executor, ...)`.
- **Обоснование:** Превентивная защита event loop от блокировок при росте объёма лидов (раздел 14.4 явно упоминает этот путь).

### D011: Email-валидатор — regex, без email-validator библиотеки
- **Дата:** 2026-06-22
- **Контекст:** Раздел 11 упоминает `email-validator` как опциональную зависимость (через `pydantic.EmailStr` либо напрямую).
- **Решение:** Реализован regex-валидатор (покрывает 99% случаев). Библиотека `email-validator` не включена в `requirements.txt`.
- **Обоснование:** MVP не требует строгой RFC-валидации. Regex проще тестировать и не добавляет зависимость. Полная RFC-валидация — будущая задача при необходимости.

### D012: Политика дубликатов — информирование пользователя
- **Дата:** 2026-06-22
- **Контекст:** Раздел 12 архитектуры: "Политика дубликатов: тихо игнорировать повторный email или сообщать пользователю" — открытый вопрос.
- **Решение:** Пользователю сообщается "Этот email уже зарегистрирован" с предложением ввести другой. FSM остаётся в `waiting_for_email`.
- **Обоснование:** Пользовательский опыт лучше с обратной связью, чем с молчаливым игнорированием. Логируется на INFO.
