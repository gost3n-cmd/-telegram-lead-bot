## 0. Резюме

Текущая система — это **генератор**: один скрипт, который при каждом запуске создаёт Excel-файл с нуля и полностью его перезаписывает. Целевая система — **генератор + бот**, где бот должен *дописывать* данные в этот же файл между запусками генератора. Это два принципиально разных режима работы с одним и тем же файлом, и именно здесь лежал главный архитектурный риск проекта.

Решение строится на четырёх принципах:

1. **Изоляция Excel за интерфейсом** — бот никогда не работает с openpyxl напрямую из хендлеров; есть слой `Storage`, который можно подменить.
2. **Один писатель за раз** — все записи бота в файл проходят через единственную точку сериализации (очередь/лок), чтобы избежать гонок при параллельных сообщениях от разных пользователей.
3. **Данные лидов живут в собственном листе** Excel-книги, физически отдельном от 11 листов контент-плана, которые создаёт `build_system.py`.
4. **Sheet Ownership — генератор владеет только своими листами.** `build_system.py` никогда не создаёт `Workbook()` вслепую и никогда не работает со списком "все листы файла" — он работает только со своим явным, фиксированным списком из 11 имён. Любой лист вне этого списка для генератора физически не существует: он не читается, не перебирается, не изменяется и не удаляется. Это и есть фикс критической проблемы (см. раздел 4).

## 2. Существующая архитектура (до этой ревизии)

Концептуально это был **однонаправленный конвейер с состоянием в памяти процесса**:

```
[хардкод-данные в коде] → [Workbook() в памяти] → [применение стилей построчно] → [wb.save() поверх файла]
```

Один процесс, один проход, нет персистентности между запусками, нет ввода извне, нет точек интеграции. Это нормальная архитектура для *генератора отчёта/шаблона*, но она была несовместима "as is" с ролью *хранилища, в которое что-то пишет внешний процесс между запусками генератора*. Раздел 4 заменяет последний и первый шаг этого конвейера ("Workbook() в памяти" → "wb.save() поверх файла"), не трогая середину.

## 3. Проблемы существующей структуры (применительно к цели проекта)

### 3.1 Перезапись файла целиком — статус: **РЕШЕНО** (см. раздел 4)

`build_system.py` всегда создавал новую `Workbook()` и сохранял её **поверх** `ai_content_os.xlsx`. Если бот дописывал лиды в этот файл, а затем кто-то повторно запускал генератор (например, чтобы обновить контент-план) — **все собранные email уничтожались**, потому что генератор не читал существующий файл и не знал о листе с лидами.

Зафиксированное решение (раздел 4, коротко):

- генератор открывает файл через `load_workbook()`, если он существует, вместо безусловного `Workbook()`;
- генератор пересоздаёт **только** свои 11 листов по явному списку `OWNED_SHEETS`;
- любой лист, которого нет в этом списке (в первую очередь — `Leads`), генератор не открывает, не читает и не удаляет ни при каких условиях.

Раздел 4 описывает это как общее архитектурное правило, а не как точечный патч, — потому что то же правило защитит и любую третью систему, которая в будущем начнёт писать в этот файл.

### 3.2 Смешение данных и презентации
Контентные данные (тексты, структуры таблиц) и логика стилизации (цвета, шрифты, ширины колонок) живут в одном файле вперемешку. Для генератора это приемлемо как есть, но это antiparttern, если бы такой же подход переиспользовался для бота. *(Вне рамок этой ревизии — не относится к перезаписи файла.)*

### 3.3 Отсутствие модульности
Всё — один файл, без пакетов, без классов, без переиспользуемых сущностей за пределами стилевых хелперов. Раздел 4 сознательно **не** решает эту проблему: исправление ограничено двумя точками входа/выхода скрипта (см. 4.7), весь остальной код остаётся как есть, по прямому требованию не делать радикальный рефакторинг.

### 3.4 Отсутствие абстракции хранилища
Запись в Excel (`wb.save(...)`) жёстко вшита в конец скрипта. Для генератора это осознанно остаётся так — у него один-единственный потребитель (файл на диске), и вводить интерфейс `StorageInterface` ради одного скрипта избыточно. Абстракция хранилища нужна и уже спроектирована для **бота** (раздел 10), потому что у бота несколько вероятных будущих реализаций (Excel → SQLite → Postgres). У генератора такой перспективы нет — он всегда пишет тот же `.xlsx`, поэтому здесь это не добавляется намеренно (см. ограничения задачи: не менять стек).

### 3.5 Нет конкурентного доступа
Скрипт запускается вручную, синхронно. `openpyxl` физически не предназначен для конкурентной записи — он работает по схеме "прочитать весь файл в память → изменить → сохранить весь файл целиком", и при двух параллельных сохранениях вторая операция перезапишет первую (lost update). **Важно: раздел 4 решает другую проблему.** Sheet Ownership гарантирует, что генератор логически не тронет данные бота, *если* их запись не пересекается по времени. Он не делает запись потокобезопасной на уровне ОС/файловой системы. Защита от буквально одновременной записи остаётся ответственностью `storage/lock.py` на стороне бота (раздел 10) и операционного правила "не запускать генератор, пока идёт активная запись бота" — это сознательно вынесено за рамки данной ревизии, см. раздел 4.6.

### 3.6 Нет обработки ошибок и логирования
Только `print()`. Подходит для разового запуска вручную; неприемлемо для долгоживущего процесса бота, где сбой записи должен быть залогирован и не должен ронять весь процесс. *(Вне рамок этой ревизии — относится к боту, не к генератору.)*

## 4. Модель безопасного взаимодействия `build_system.py` с Excel-файлом

Это центральный раздел ревизии. Он отвечает на один вопрос: **как генератор должен открывать и сохранять файл, чтобы никогда не уничтожить данные, которые он не создавал.**

### 4.1 Архитектурное правило

Excel-файл `data/ai_content_os.xlsx` — это **общее хранилище (shared storage) между двумя независимыми системами**:

- **System A: Content Generator** (`build_system.py`) — пишет 11 листов контент-плана.
- **System B: Telegram Bot** (`storage/excel_storage.py`) — пишет один лист `Leads`.

Эти системы не имеют программных зависимостей друг от друга (см. раздел 9) и **не должны пересекаться логически**: ни одна из них не имеет права читать, изменять или удалять листы, принадлежащие другой. Связь между ними — только через факт совместного использования одного файла на диске, и эта связь регулируется не кодом одной системы, контролирующим другую, а **симметричным правилом**, которому следует каждая система в отношении своих собственных листов.

Для генератора это правило реализуется как **Sheet Ownership**.

### 4.2 Sheet Ownership — список владения

Генератор явно объявляет, какими листами он владеет. Это не "всё, что не Leads", а позитивный список — так правило остаётся верным, даже если в будущем появится System C со своим листом, о котором генератор никогда не узнает.

```
OWNED_SHEETS = [
    "🏠 Home",
    "🎯 01 Strategy",
    "📱 02 Accounts",
    "🧠 03 Psychology",
    "🎬 04 Content Engine",
    "⚙️ 05 Pipeline",
    "👥 06 Team",
    "🛠️ 07 AI Stack",
    "📈 08 KPI Dashboard",
    "📚 09 Content Bank",
    "🔬 10 Reality Check",
]
```

Правило: **этот список — единственный источник истины о том, какие листы вправе создавать, пересоздавать или удалять генератор.** Любое имя листа, отсутствующее в нём (в первую очередь — `Leads`, но и любое другое, появившееся независимо), для генератора как будто не существует.

Эксплуатационное следствие: имена в `OWNED_SHEETS` — зарезервированные имена. Никакая другая система (включая бота) не должна называть свой лист одним из этих 11 имён — иначе при следующем запуске генератора он будет на законных основаниях пересоздан как "свой".

### 4.3 Алгоритм: read-if-exists → rebuild owned → preserve foreign → save

1. **Файла нет на диске** → поведение как раньше: `wb = Workbook()`, все 11 листов строятся с нуля той же логикой, что и сейчас.
2. **Файл существует** →
   a. `wb = load_workbook(OUT_PATH)` — в память загружаются **все** существующие листы, включая чужие (например, `Leads`), со всем их содержимым и позицией.
   b. Для каждого имени из `OWNED_SHEETS`: если лист с таким именем уже есть в книге — он удаляется (`wb.remove(wb[name])`). Удаляется только этот один лист, ничего больше.
   c. Каждый из 11 листов строится заново той же существующей функцией, что и сейчас (`wb.create_sheet(name)` + неизменная логика наполнения).
   d. Ни один лист, имя которого не входит в `OWNED_SHEETS`, ни разу не передаётся ни в один из хелперов (`cl()`, `mspan()`, `section()` и т.д.) и не упоминается ни в одном вызове `wb[...]`. Это не флаг "пропустить при записи" — это структурное отсутствие кода, который мог бы к нему обратиться.
3. **Восстановление порядка вкладок** (косметика, не затрагивает содержимое): 11 листов из `OWNED_SHEETS` переставляются в начало книги в каноническом порядке; все чужие листы остаются после них в том порядке, в котором уже были. Порядок листов на содержимое не влияет — это только удобство навигации в Excel.
4. `wb.save(OUT_PATH)` — тот же вызов, что и раньше, но теперь сохраняется книга, где 11 листов свежесобраны, а все чужие листы (в том числе `Leads` со всеми лидами) physически — те же объекты, что были загружены в шаге (a), ни разу не модифицированные.

### 4.4 Псевдокод (архитектурный уровень, не реализация)

```
OWNED_SHEETS = [...]                      # см. 4.2, 11 точных имён

if not path_exists(OUT_PATH):
    wb = Workbook()
    # построить все 11 листов с нуля — без изменений в существующей логике
else:
    wb = load_workbook(OUT_PATH)
    for name in OWNED_SHEETS:
        if name in wb.sheetnames:
            wb.remove(wb[name])
    # построить все 11 листов заново той же логикой —
    # wb.create_sheet(name) для каждого имени из OWNED_SHEETS,
    # дальше код наполнения листов не меняется ни на строку

reorder_owned_first(wb, OWNED_SHEETS)     # косметика: переставить 11 листов в начало
wb.save(OUT_PATH)
```

Финальная реализация (точные вызовы `wb.remove`, обработка кириллических/emoji-имён, выбор способа переупорядочивания листов в установленной версии `openpyxl`) — задача Этапа 2, не этой ревизии.

### 4.5 Граничные случаи

| Ситуация | Поведение генератора |
|---|---|
| Файла нет вообще (первый запуск проекта) | Создаёт файл с нуля, 11 листов. Листа `Leads` ещё нет — это нормально, его создаст бот при первом сохранённом лиде. |
| Файл есть, `Leads` ещё нет (генератор запускался, бот — ни разу) | Пересобирает свои 11 листов, не создаёт `Leads` сам — это не его ответственность (раздел 9, `ExcelStorage`). |
| Файл есть, в `Leads` уже 10 000 строк | Лист `Leads` загружается в память как есть в шаге 4.3(a) и сохраняется как есть — генератор его не открывает и не итерируется по нему, поэтому объём данных в нём никак не влияет на работу генератора. |
| Кто-то вручную переименовал лист `Leads` в `🏠 Home` | Лист будет на следующем запуске генератора удалён и пересоздан как часть owned-набора — это ожидаемое поведение зарезервированных имён (см. 4.2), а не баг. |
| Бот добавил 12-й лист с именем, случайно совпадающим с одним из `OWNED_SHEETS` | Тот же случай, что выше — имя зарезервировано, совпадение интерпретируется как "это мой лист". Эксплуатационное правило, а не исключение из кода. |

### 4.6 Что эта модель **не** решает

- **Конкурентную запись в буквальном смысле** — если генератор и бот запишут файл в один и тот же момент времени, выиграет тот, кто сохранит последним, независимо от Sheet Ownership (см. 3.5). Это решается на уровне `storage/lock.py` бота и операционного правила запуска, не на уровне генератора.
- **Целостность данных внутри `Leads`** — генератор гарантирует только то, что не тронет этот лист; за корректность его содержимого по-прежнему отвечает исключительно `ExcelStorage`.
- **Миграцию на другое хранилище** — Sheet Ownership остаётся актуальной только пока обе системы используют один `.xlsx`-файл; при переходе бота на SQLite/Postgres (раздел 11, вне рамок) этот раздел перестаёт быть нужен для бота, но продолжает быть нужен генератору ровно в том же виде, пока он сам пишет в Excel.

### 4.7 Размещение в коде

Изменение **минимально по площади**: это не новый модуль и не новый слой архитектуры, а две точки внутри уже существующего `build_system.py`:

- **В начале скрипта** — заменить безусловное `wb = Workbook()` на проверку существования файла + `load_workbook()`/`Workbook()` + удаление 11 owned-листов при необходимости (4.3, шаги 1–2).
- **В конце скрипта** — между текущим построением листов и текущим `wb.save(OUT_PATH)` добавить один шаг переупорядочивания (4.3, шаг 3).

Всё, что между этими двумя точками — 1500 строк построения 11 листов — **не меняется**. Список `OWNED_SHEETS` логично разместить рядом с остальными константами в начале файла (туда же, где уже определены цвета `INK`, `ACCENT` и т.д.), чтобы он был в одном месте с именами листов, которые присваиваются им чуть ниже (`ws0.title = "🏠 Home"` и т.д.) — это минимизирует риск рассинхронизации списка с реальными именами.

## 5. Новая архитектура проекта

Целевая система — это **два независимых процесса, которые делят один файл как общий ресурс** (раздел 4), плюс новый бот, построенный на классической слоистой архитектуре с инверсией зависимостей (Dependency Inversion) на границе хранилища.

Слои бота (сверху вниз — от входящего апдейта до диска):

1. **Presentation** — aiogram-хендлеры, FSM-состояния, клавиатуры. Знает про Telegram, ничего не знает про Excel.
2. **Application (Service)** — оркестрирует сценарий "принять email": вызывает Validation, затем Storage, формирует ответ. Не знает про openpyxl и не знает про aiogram-объекты напрямую (получает на вход уже примитивы/DTO).
3. **Domain** — модель `Lead` и **интерфейс** `StorageInterface`. Это "контракт", от которого зависят и Application, и конкретные реализации хранилища. Сам по себе не зависит ни от чего.
4. **Infrastructure** — конкретная реализация хранилища (`ExcelStorage` сейчас, `SQLiteStorage`/`PostgresStorage` в будущем), а также механизм безопасной последовательной записи (лок/очередь).

Ключевое архитектурное решение: **Application зависит от интерфейса (Domain), а не от конкретной реализации (Infrastructure)**. Конкретная реализация подставляется один раз — в точке композиции (`main.py`) — и больше нигде в коде не упоминается напрямую. Это и есть механизм, который позволяет позже заменить Excel на БД, не трогая хендлеры и сервисы.

Существующий генератор (`build_system.py`) остаётся отдельным, самостоятельным скриптом — он не становится частью пакета бота и не импортируется ботом. Связь между ними — только на уровне общего файла на диске, и с этой ревизии эта связь больше не является источником риска потери данных (раздел 4).

## 8. Поток данных

Telegram User → Telegram Bot → Validation Layer → Storage Layer → Excel

Ниже — развёрнутая версия с веткой ошибки валидации.

*(см. диаграмму ниже в чате)*

Текстовое описание шагов:

1. Пользователь отправляет боту сообщение с email (в ответ на запрос в рамках FSM-сценария).
2. Хендлер (`handlers/email_collection.py`) получает текст и передаёт его в `lead_service`, не пытаясь сам его валидировать или сохранять.
3. `lead_service` вызывает `email_validator` — проверка синтаксиса и нормализация.
   - Если email невалиден — сервис сразу возвращает хендлеру отрицательный результат, **Storage Layer не вызывается вообще**. Хендлер просит пользователя ввести email заново.
4. Если синтаксис валиден, `lead_service` обращается к `StorageInterface.email_exists()` — проверка дубликата как бизнес-правило (живёт в Application, а не в Storage, потому что это решение "что считать дубликатом" — часть бизнес-логики, а не деталь хранения).
5. `lead_service` вызывает `StorageInterface.save_lead(lead)`.
6. `ExcelStorage` встаёт в очередь/захватывает лок (раздел 3.5), открывает файл, находит лист `Leads`, дописывает строку, сохраняет файл, освобождает лок.
7. Результат (успех/ошибка хранения) возвращается вверх по цепочке: Storage → Service → Handler → пользователю отправляется подтверждение или сообщение об ошибке.

Для генератора отдельный поток данных описан в разделе 4.3 — он не пересекается с этим потоком ни в одной точке, кроме физического файла на диске.

## 9. Схема зависимостей между модулями

*(см. диаграмму ниже в чате)*

Кратко в виде правил:

- `handlers/*` зависят от `services/lead_service.py`, `states/*`, `keyboards/*`. Никогда не импортируют `storage/excel_storage.py` напрямую.
- `services/lead_service.py` зависит от `validation/email_validator.py` и от `storage/base.py` (интерфейса), но не от `storage/excel_storage.py`.
- `storage/excel_storage.py` зависит от `storage/base.py` (реализует контракт), `storage/lock.py`, `models/lead.py`. О существовании `lead_service.py` ничего не знает.
- `models/lead.py` — лист без исходящих зависимостей внутри проекта; на него ссылаются и Application, и Storage.
- `main.py` — единственный модуль, который импортирует одновременно `storage/excel_storage.py` (конкретный класс) и `storage/base.py` (интерфейс), чтобы связать их в момент старта приложения.
- `generator/build_system.py` не имеет программных зависимостей от пакета `bot/` и наоборот — связь только через файл `data/ai_content_os.xlsx` на уровне диска (пунктирная связь на диаграмме). До этой ревизии это создавало риск перезаписи (бывший раздел 3.1); теперь риск устранён моделью Sheet Ownership (раздел 4) — генератор структурно не способен затронуть данные, записанные ботом, независимо от того, в каком порядке и сколько раз запускались эти два процесса.

## 10. Интерфейс Storage Layer

Контракт спроектирован так, чтобы `ExcelStorage`, будущий `SQLiteStorage` и будущий `PostgresStorage` реализовывали один и тот же набор методов, и `lead_service.py` не заметил разницы.

| Метод | Параметры | Возвращает | Назначение |
|---|---|---|---|
| `save_lead` | `lead: Lead` | `SaveResult` (`success: bool`, `error: str \| None`) | Атомарно добавляет одну запись в хранилище |
| `email_exists` | `email: str` | `bool` | Проверка уникальности перед сохранением (используется сервисом до `save_lead`) |
| `count_leads` | — | `int` | Текущее количество сохранённых записей — для статистики/админ-команд бота |
| `list_leads` *(опционально, не нужен для MVP)* | `limit: int`, `offset: int` | `list[Lead]` | Постраничная выборка — задел под будущую админ-панель или экспорт |
| `health_check` | — | `bool` | Проверка доступности хранилища (файл не залочен сторонним процессом / БД отвечает) — полезно для команды `/status` или мониторинга |

Модель данных `Lead` (поля, не код):

| Поле | Тип | Комментарий |
|---|---|---|
| `telegram_id` | int | Уникальный идентификатор пользователя в Telegram |
| `telegram_username` | str \| None | Может отсутствовать у пользователя |
| `email` | str | Уже нормализованный (lowercase, без пробелов) на момент попадания в Storage |
| `submitted_at` | datetime (UTC) | Момент сохранения |
| `source` | str | Например, `"bot_v1"` — задел на случай нескольких каналов сбора email в будущем |

Принцип, который делает замену хранилища безопасной: **все методы интерфейса оперируют только примитивами и моделью `Lead`/`SaveResult` — ничего специфичного для Excel (номера строк, листы, ячейки) не просачивается наружу через контракт.** Всё, что знает про openpyxl, инкапсулировано внутри `excel_storage.py` и никогда не пересекает границу интерфейса.

## 13. FSM-сценарий сбора email

Бот управляет диалогом через конечный автомат (Finite State Machine) `aiogram.fsm`. FSM нужна, поскольку сбор email — это многошаговый диалог: без фиксации состояния хендлер не мог бы различить "пользователь ввёл email в ответ на запрос" и "пользователь написал что-то в произвольный момент".

### 13.1 Состояния

```python
class EmailStates(StatesGroup):
    waiting_for_email = State()
```

Для MVP достаточно одного состояния. Переход в него выполняется хендлером `/start`. Хендлер `email_collection.py` принимает входящий текст **только** если пользователь находится в `EmailStates.waiting_for_email` — фильтрация по состоянию задаётся на уровне роутера aiogram, не внутри функции хендлера.

### 13.2 Переходы состояний

```
                /start
[нет состояния] ─────────────────────────────────────► [waiting_for_email]
                                                                │
        ┌───────────────────────────────────────────────────────┘
        │  входящий текст (F.text) в состоянии waiting_for_email
        ▼
email_validator.validate(raw_text)
       / \
 невалид   валид + нормализован
    │              │
    │              ▼
    │      lead_service.email_exists(email)
    │              / \
    │           да    нет
    │           │       │
    │           │       ▼
    │           │   lead_service.save_lead(lead)
    │           │          / \
    │           │       ошибка   ok
    │           │          │       │
    ▼           ▼          ▼       ▼
[waiting_for_email сохраняется]   [FSM сброшена → нет состояния]
    │           │          │               │
"введи      политика    "ошибка,       "email
снова"      (раздел     попробуй       сохранён"
             12)        позже"
```

### 13.3 Граничные случаи FSM

| Ситуация | Ожидаемое поведение |
|---|---|
| Пользователь повторно вызывает `/start` в состоянии `waiting_for_email` | Хендлер `/start` сбрасывает FSM и запускает сценарий заново — пользователь не блокируется |
| Пользователь отправляет нетекстовый контент (фото, стикер, файл) | Хендлер `email_collection.py` фильтрован на `F.text`; нетекстовые апдейты до него не доходят — обрабатываются другим хендлером или игнорируются |
| Пользователь впервые пишет боту без `/start` | `waiting_for_email` не установлено; `email_collection.py` не сработает; имеет смысл добавить fallback-хендлер, подсказывающий `/start` |
| Пользователь долго не отвечает, затем пишет email | Состояние FSM сохраняется без таймаута (MVP). Таймаут не вводится без явного требования к нему |

## 14. Механизм сериализации записи (storage/lock.py)

### 14.1 Проблема

`ExcelStorage.save_lead()` работает по схеме: **загрузить весь файл → изменить в памяти → сохранить весь файл**. При одновременном поступлении двух Telegram-апдейтов (два пользователя пишут email параллельно) оба хендлера запустят `save_lead()` конкурентно в рамках одного `asyncio`-event loop. Без синхронизации второй `load_workbook` прочитает файл раньше, чем первый `wb.save` завершится — одна из записей будет потеряна (lost update).

### 14.2 Решение: asyncio.Lock

Бот работает в одном процессе с одним event loop, поэтому для сериализации записи достаточно `asyncio.Lock` — примитива стандартной библиотеки Python, отдельной зависимости не требует.

Лок создаётся один раз как атрибут `ExcelStorage` при инициализации. Инстанция `ExcelStorage` создаётся в `main.py` и передаётся через DI — это гарантирует, что в пределах процесса существует ровно один экземпляр лока.

```python
# conceptual — не финальная реализация
class ExcelStorage(StorageInterface):
    def __init__(self, file_path: str):
        self._file_path = file_path
        self._lock = asyncio.Lock()
```

`asyncio.Lock` работает кооперативно: второй `save_lead`, встретив занятый лок, отдаёт управление event loop и возобновляется после освобождения — event loop при этом не блокируется.

### 14.3 Граница критической секции

Лок удерживается на всём промежутке `load_workbook → modify → save`. Освобождение лока до `wb.save` воспроизвело бы ту же гонку.

```
# псевдокод — граница критической секции

async def save_lead(self, lead: Lead) -> SaveResult:
    async with self._lock:                          # ← захват лока
        wb = load_workbook(self._file_path)         # читаем весь файл
        ws = _ensure_leads_sheet(wb)                # находим или создаём Leads
        ws.append(_lead_to_row(lead))               # дописываем строку
        wb.save(self._file_path)                    # сохраняем весь файл
                                                    # ← освобождение лока
```

Если `email_exists()` реализована как отдельный `load_workbook`, её вызов должен либо входить в ту же критическую секцию, либо сопровождаться кешем прочитанных email в памяти (инвалидация после каждого успешного `save_lead`). Конкретный выбор — задача Этапа 2.

### 14.4 Ограничения

| Ограничение | Следствие |
|---|---|
| Только внутри одного OS-процесса | `asyncio.Lock` не синхронизирует два отдельных процесса. Если бот запущен дважды или масштабируется горизонтально — нужен file-level lock (`filelock` / `portalocker`, раздел 11) |
| Генератор не участвует в локе бота | Буквально одновременный запуск `build_system.py` и `save_lead` бота — операционный риск (раздел 4.6), `asyncio.Lock` его не устраняет |
| `openpyxl` блокирует event loop | `load_workbook` и `wb.save` — синхронный disk I/O, они занимают event loop на время операции. При малом объёме лидов несущественно; при необходимости решается через `asyncio.get_event_loop().run_in_executor(None, ...)` |

## 15. Стратегия обработки ошибок

### 15.1 Таксономия ошибок

| Тип ошибки | Источник | Класс / возврат | Что видит пользователь | Что логируется |
|---|---|---|---|---|
| Невалидный синтаксис email | `email_validator` | возврат `False`, без исключения | "Email не распознан — введи ещё раз"; FSM остаётся в `waiting_for_email` | DEBUG: невалидный ввод, `telegram_id` |
| Дубликат email | `lead_service` через `email_exists()` | возврат `bool`, без исключения | Определяется политикой (раздел 12) | INFO: попытка дубликата, `telegram_id`, нормализованный email |
| Хранилище недоступно (файл занят, нет прав, нет места) | `ExcelStorage` | `StorageUnavailableError` | "Что-то пошло не так — попробуй позже" | ERROR: полный traceback, `telegram_id` |
| Непредвиденная ошибка при записи | `ExcelStorage` | `StorageError` | То же "попробуй позже" | ERROR: полный traceback, `telegram_id`, тип исключения |
| Повреждение Excel-файла (`BadZipFile` и аналоги) | `ExcelStorage` при `load_workbook` | перехватывается как `StorageUnavailableError` | То же "попробуй позже" | ERROR: тип исключения openpyxl, путь к файлу |

### 15.2 Принцип распространения ошибок через слои

```
Infrastructure: ExcelStorage
    выбрасывает → StorageUnavailableError | StorageError
        │
        ▼
Application: lead_service
    перехватывает StorageUnavailableError, StorageError
    преобразует → ServiceResult(success=False, error_type=ServiceErrorType.STORAGE_UNAVAILABLE)
    НЕ пробрасывает Storage-исключения наружу
        │
        ▼
Presentation: handler
    получает ServiceResult
    выбирает текст сообщения по error_type
    НЕ импортирует StorageUnavailableError, не знает про openpyxl
```

Следствие инверсии зависимостей (раздел 5): если `ExcelStorage` заменяется на `SQLiteStorage` с другим набором исключений — Presentation это не затрагивает; `lead_service` может потребовать точечного обновления маппинга ошибок.

### 15.3 Что не обрабатывается в коде

- **Одновременная запись генератором**: если `build_system.py` сохраняет файл в момент незавершённого `save_lead` бота — ни исключение, ни `asyncio.Lock` бота эту ситуацию не поймает. Операционный риск, раздел 4.6.
- **Потеря Telegram-апдейта при downtime бота**: апдейт не придёт повторно после 24ч при long polling. Вне рамок данной архитектуры.
- **Восстановление повреждённого xlsx**: обнаруживается при следующем `load_workbook` и ловится как `StorageUnavailableError`; автоматического восстановления нет — ручная операция администратора.

## 16. Схема конфигурации (.env)

`config.py` читает переменные окружения через `pydantic-settings`. Объект конфигурации создаётся один раз в `main.py` и передаётся в `ExcelStorage` и `Bot` через DI — нигде больше не импортируется напрямую.

| Переменная | Тип | Обязательна | Дефолт | Назначение |
|---|---|---|---|---|
| `BOT_TOKEN` | `str` | да | — | Telegram Bot API token. Не логируется, исключена из `repr` объекта конфигурации (`pydantic.SecretStr`) |
| `EXCEL_FILE_PATH` | `str` | нет | `data/ai_content_os.xlsx` | Путь к общему Excel-файлу. Может быть абсолютным или относительным к рабочей директории процесса |
| `LEADS_SHEET_NAME` | `str` | нет | `Leads` | Имя листа бота в Excel-книге. **Не должно совпадать** ни с одним из 11 имён в `OWNED_SHEETS` генератора (раздел 4.2) — иначе при следующем запуске генератор удалит лист как свой |
| `LOG_LEVEL` | `str` | нет | `INFO` | Уровень логирования: `DEBUG` / `INFO` / `WARNING` / `ERROR` |
| `ENVIRONMENT` | `str` | нет | `production` | `production` или `development`. Используется для управления детализацией логирования; при необходимости — для подстановки пути к тестовому файлу |

`LEADS_SHEET_NAME` сделана конфигурируемой не ради гибкости как таковой, а чтобы сделать имя листа **явной**, а не молчаливой константой: это точка сопряжения между двумя независимыми системами, и её изменение должно быть осознанным действием в одном месте.

Конфигурация `build_system.py` не выносится в `.env`. Генератор не имеет runtime-параметров; путь к выходному файлу задаётся константой `OUT_PATH` в коде. Параметризация генератора, если она когда-либо понадобится, — отдельная задача, не связанная с архитектурой бота.

### А.2 Схема зависимостей (к разделу 9)

Легенда: `──►` прямой импорт; `- -►` связь только через файл на диске (не импорт)

  ├──► config.py
  │

  │         └──► keyboards/common.py
  │
  ├──► handlers/email_collection.py
  │         ├──► states/email_states.py
  │         ├──► keyboards/common.py

  │                   │         └──► models/lead.py
  │                   ├──► storage/base.py  (StorageInterface)
  │                   │         └──► models/lead.py
  │                   └──► models/lead.py
  │
  ├──► storage/excel_storage.py   ◄── конкретную реализацию знает только main.py
  │         ├──► storage/base.py  (реализует StorageInterface)
